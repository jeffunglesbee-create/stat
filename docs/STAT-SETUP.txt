STEP 1 — PUSHOVER SETUP ($4.99 one-time, 30-day free trial)
═══════════════════════════════════════════════════════════════

Pushover delivers high-priority alerts that bypass Do Not Disturb.
This is how P1 (Epic/EHR) job matches reach you instantly.

1. Download Pushover app: iOS App Store or Google Play
2. Create account at pushover.net
3. Note your User Key on the dashboard
4. Create application at pushover.net/apps/build

   - Type: Application
   - Icon: (optional)

You will use both values in Step 4 (Worker secrets).

Push priority behavior:
  P1 matches → priority=1 → siren sound, bypasses quiet hours
  P2 matches → priority=0 → normal Pushover sound

STEP 5 — CREATE KV NAMESPACE
═══════════════════════════════════════════════════════════════

From your local terminal (wrangler must be authenticated):

  wrangler login    (if not already authenticated)
  wrangler kv:namespace create STAT_KV

Copy the returned id value, e.g.:
  Add the following to your configuration file in your kv_namespaces array:
  { binding = "STAT_KV", id = "abc123...def456" }

Edit wrangler.toml in the stat-repo:
  Find: id = "REPLACE_WITH_YOUR_KV_NAMESPACE_ID"
  Replace with the actual ID from the output above

Commit and push:
  git add wrangler.toml
  git commit -m "Set KV namespace ID"
  git push

This will trigger the CI deploy.

STEP 6 — SET WORKER SECRETS
═══════════════════════════════════════════════════════════════

These are the notification credentials. They live ONLY in Cloudflare
Workers — never in GitHub, never in wrangler.toml, never in git.

IMPORTANT: wrangler deploy does NOT overwrite secrets.
Set them once. They persist across all future code deploys forever.
When a secret needs to rotate, run the command again.

From your local terminal (after wrangler login):

  wrangler secret put PUSHOVER_APP_TOKEN --name stat-job-watcher
  [Enter the Pushover API Token from Step 1]

  wrangler secret put PUSHOVER_USER_KEY --name stat-job-watcher
  [Enter the Pushover User Key from Step 1]

  wrangler secret put SENDGRID_API_KEY --name stat-job-watcher
  [Enter the SendGrid API key from Step 2]

  wrangler secret put ALERT_EMAIL --name stat-job-watcher
  [Enter the email address to receive alert digests]

Verify all are set:
  wrangler secret list --name stat-job-watcher

Expected output:
  PUSHOVER_APP_TOKEN
  PUSHOVER_USER_KEY
  SENDGRID_API_KEY

STEP 8 — BOOTSTRAP (first deploy only)
═══════════════════════════════════════════════════════════════

This primes all the Durable Objects and salary caches:

  curl -X POST https://stat-job-watcher.<subdomain>.workers.dev/bootstrap

Replace <subdomain> with your Workers subdomain
(found at: dash.cloudflare.com → Workers & Pages → Overview → 
 click stat-job-watcher → the URL shown is the Worker URL)

Expected response:
  {"ok":true,"spawned":50,"total":50}  (approximate numbers)

Bootstrap:
  - Spawns CompanyWatcherDO for all ~50 seed companies
  - Starts BLS OEWS cache refresh (async, ~10s)
  - Starts DOL LCA download + parse (async, ~30–60s)

DOs begin polling immediately after bootstrap.
First alerts may arrive within 1–2 minutes if matching jobs exist.

Verify salary caches after ~2 minutes:
  curl https://stat-job-watcher.<subdomain>.workers.dev/salary-status

Expected (after bootstrap):
  {"peerCount":0,"lcaCount":50000+,"blsDate":"2026-...","lcaDate":"2026-..."}
  (lcaCount will vary; blsDate and lcaDate confirm caches populated)

ONGOING OPERATIONS
═══════════════════════════════════════════════════════════════

ADDING A NEW COMPANY:
  Fastest (no redeploy):
    curl -X POST https://stat-job-watcher.*.workers.dev/companies \
      -H "Content-Type: application/json" \
      -d '{"name":"Acme Health","ats":"greenhouse","token":"acmehealth"}'
  
  Or via UI: Companies tab → + Add company → fill form → submit

  Permanent (persists through redeploys):
    1. Add to SEED_COMPANIES in src/config.js
    2. git add src/config.js && git commit -m "Add Acme Health to watchlist"
    3. git push (triggers CI deploy)
    4. POST /bootstrap (spawns DO for new company)

UPDATING KEYWORDS:
  1. Edit WATCH_GROUPS in src/config.js
  2. git add src/config.js && git commit -m "Update keyword groups"
  3. git push (triggers CI deploy)
  Note: keyword matching runs on the Worker, not in DOs.
  No DO restart needed. Updated keywords take effect after next
  alarm cycle for each company.

REFRESHING SALARY CACHES:
  BLS OEWS updates annually (May):
    curl -X POST https://stat-job-watcher.*.workers.dev/salary-refresh

  DOL LCA updates quarterly (Q1=Jan, Q2=Apr, Q3=Jul, Q4=Oct):
    Same command — /salary-refresh re-fetches both BLS and LCA.

  Recommended schedule: POST /salary-refresh on the first of each
  month. The command is idempotent — safe to run anytime.

CLEARING SEEN IDs (re-alert on all current jobs):
  Use case: want to see all currently live matching jobs,
  e.g. after changing keyword groups.

  curl -X POST https://stat-job-watcher.*.workers.dev/reset-seen
  
  CAUTION: This will generate alerts for ALL currently live
  matching jobs on the next poll cycle. Could be noisy.

CHECKING STATUS:
  System overview:        GET /
  Companies + DO status:  GET /companies
  Salary cache status:    GET /salary-status
  Specific DO status:     GET /do/greenhouse:nordicglobal/status

NUCLEAR RESET (start over):
  curl -X POST https://stat-job-watcher.*.workers.dev/reset-all
  # Then re-bootstrap:
  curl -X POST https://stat-job-watcher.*.workers.dev/bootstrap

RESUME MATCHER USAGE
═══════════════════════════════════════════════════════════════

The resume-matcher.html file uses the Anthropic API to score
job descriptions against your resume profile.

IMPORTANT: The Anthropic API call goes to api.anthropic.com from the
browser. This works in Claude.ai's artifact viewer but NOT when
opening the file locally from disk (browser CORS blocks it).

Options for using resume-matcher.html:
  1. Upload to Claude.ai as an artifact and interact there
  2. Host on any simple web server (even python -m http.server)
  3. Open via Claude.ai's "Open in Claude" feature if available

Workflow:
  1. Drop your resume PDF or paste resume text
  2. Click "Analyze Resume" — Claude extracts a structured profile
  3. Edit the profile if needed (JSON edit button)
  4. Paste any job description → "Score this job →"
  5. Get: 1-10 score, verdict, strengths, gaps, salary alignment,
     and a tailored cover letter opener (2 sentences, specific to the
     actual job and your actual background)

The cover letter opener: copy button provided. Two sentences:
  - References specific job requirements (not generic phrases)
  - Connects to your specific Epic modules or experience
  - More effective than any template opener

TROUBLESHOOTING
═══════════════════════════════════════════════════════════════

"Worker not found" after deploy:
  Check: wrangler.toml name = "stat-job-watcher" is correct
  Check: CI deploy succeeded (GitHub Actions tab)
  Check: Cloudflare account ID in deploy.yml is b57e9af57ab46c52ca9215804e689c29

No alerts despite matching jobs:
  1. GET / — confirm activeDOs > 0
  2. GET /companies — confirm companies are marked "watching"
  3. POST /trigger — force a HiringCafe scan
  4. POST /reset-seen — re-alert all current jobs (noisy!)
  5. Check Pushover API token validity at pushover.net
  6. Check SendGrid API key at sendgrid.com

KV namespace errors:
  "Error: KV namespace not found" — KV ID in wrangler.toml is wrong
  Run: wrangler kv:namespace list
  Find the STAT_KV namespace and use its correct ID

DO migration errors:
  If you add a new DO class or modify migrations tag:
  Error: "Cannot apply migration ... without deleting existing data"
  Solution: The v1 migration covers both DO classes.
  Only add a new migrations entry if adding NEW DO classes in future.

Worker secret missing in runtime:
  Error behavior: Pushover/email silently fails, no crash
  Diagnosis: wrangler secret list --name stat-job-watcher
  Fix: wrangler secret put [SECRET_NAME] --name stat-job-watcher

SECURITY NOTES
═══════════════════════════════════════════════════════════════

Worker secrets (PUSHOVER_APP_TOKEN, PUSHOVER_USER_KEY,
SENDGRID_API_KEY, ALERT_EMAIL):
  - Set via wrangler secret put (CLI only)
  - Never in git, never in wrangler.toml, never in CI logs
  - Persist across deploys forever
  - Visible in CF dashboard under Worker settings (name only, not value)
  - To rotate: run wrangler secret put again with new value

CLOUDFLARE_API_TOKEN (GitHub secret):
  - Stored in GitHub Actions secrets (encrypted at rest)
  - Never appears in CI logs
  - Scoped to Workers:Edit only (not full CF account access)
  - To rotate: generate new token in CF dashboard, update GitHub secret

HTTP endpoints (GET /, GET /companies, etc.):
  - No authentication on HTTP endpoints currently
  - Worker URL is public but unguessable unless shared
  - If security is a concern: add a simple secret header check
    in handleFetch() in src/index.js

Resume matcher (resume-matcher.html):
  - Runs entirely in browser
  - Resume data sent to Anthropic API only (not to STAT Worker)
  - No resume data stored anywhere
  - Job scoring: only job description and extracted profile sent to API
